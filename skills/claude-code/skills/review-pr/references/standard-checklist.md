# 标准模式 — 完整审查 Checklist

## 全局：设计文档对照

如 Step 2 获取到设计文档，**所有章节**均须对照设计文档审查，不限于 2.1：
- 实现是否与设计意图一致？偏离处标注原因（合理扩展 / 越界 / 遗漏）
- 设计中明确提及的功能是否都已实现？
- 实现中是否包含设计未描述的额外行为？

未获取到设计文档时，基于代码本身推断意图，各章节正常执行。

## 〇、总结（TL;DR）

30 秒内让读者掌握全貌：
- 一句话概括 PR 做了什么
- 发现的问题统计（🔴 / 🟡 / 🟢 各多少条）
- 🔴 必须修改的问题摘要列表（来源不限于代码审查，涵盖所有章节）
- PR 描述质量：一句话评价
- 合并建议（建议合并 / 修复后合并 / 需要重大修改）

**🔴 问题来源说明**：必须修改的问题可来自任何章节——代码审查（bug/逻辑错误）、方案评审（设计不合理）、测试评审（UT/BVT 不达标、回归复现不充分）、风险检查（安全/并发问题）。只要达到 🔴 级别（不修就不能合并），都列入摘要。

## 一、PR 描述评审

### 设计文档状态

首先标注设计文档获取结果：
- ✅ **已获取设计文档** — 来源：PR body / Issue #xxx / PR comment（一句话概括方案要点）
- ⚠️ **未找到设计文档** — PR body、关联 Issue、PR comments 均未包含方案描述。建议作者补充设计说明。

### PR body 四维评审

对 PR body（及关联 Issue，如有）按四维结构评审：

**背景（Context）**
- 是否清晰描述当前现状和上下文？
- 读者能否仅凭描述理解问题所处的业务/技术场景？

**原因（Why）**
- 是否说明为什么要做？驱动力是什么？
- 问题的严重程度和影响范围是否有数据支撑？

**方案（How）**
- 是否描述了具体的技术方案或实现思路？
- 方案是否足够详细，能指导 Review？
- 是否讨论了备选方案及取舍理由？

**结果（Expected Outcome）**
- 是否明确预期结果和验收标准？
- 是否有可衡量的成功指标？

输出格式：
- ✅ 充分 + 简要说明
- 🟡 不足 + 缺什么 + 建议补充
- 🔴 缺失 + 为什么需要

## 二、方案评审

### 2.0 变更可视化（状态机 / 调用链 / 数据流图）

用图示说明本 PR 实际修改了什么。根据变更性质选择最合适的表达方式：

**选择依据**：
- 状态迁移/生命周期变更 → 状态机图
- 函数调用路径变更 → 调用链图
- 数据在模块间流转方式变更 → 数据流图
- 多种混合 → 选最能体现核心修改的一种，必要时组合

**输出格式**（ASCII 图 + 标注）：

```
修改前（简化）：
A → B → C → D
         ↑ 问题点

修改后：
A → B ──→ D     ← 移除 C 对 B 的依赖
     ↘ C（独立）  ← NEW / ← 修改
```

**标注规则**：
- `← NEW`：新增路径/节点
- `← 修改`：已有节点行为变更
- `← 移除`：删除的路径/节点（用删除线或注释）
- `← 问题点`：修复前的故障位置

**要求**：
- 图必须基于 diff 中实际代码变更绘制，不能是对 PR 描述的复述
- 涉及条件分支时标注分支条件
- 复杂场景可分"修改前"和"修改后"两张图对比

### 2.1 方案合理性

从三个维度综合评估：

**A. 函数调用场景**
- 变更函数在实际调用链中的位置和角色
- 调用方（caller）是否受到影响？是否需要感知此变更？
- 被调用方（callee）的契约是否被破坏？
- 是否有更简洁的实现方式（更少的调用层级、更直接的路径）？
- 是否有可复用的现有组件/库被忽略？

**B. 使用场景覆盖**
- 列出该变更涉及的所有使用场景（正常流、异常流、边界场景）
- 每个场景下方案是否正确处理？
- 是否存在未覆盖的场景会导致回归？
- 向后兼容性：现有调用方在不修改代码的情况下行为是否一致？

**C. 设计决策评审**（重构/链路变更时必填，简单 fix 可跳过）

当 PR 涉及链路重构、算法替换、协议变更等设计决策时，逐条列出关键决策并评估：

| 设计决策 | 旧方案 | 新方案 | 收益 | 引入风险 | 判定 |
|----------|--------|--------|------|----------|------|
| 例：坐标系 | 像素坐标 | 0-1000 归一化 | VLM 输出一致性↑ | 旧 prompt 不兼容 | ✅ 合理 |

评估维度：
- 新方案的收益是否能覆盖引入的风险？
- 是否有更简单的替代方案被忽略？
- 决策之间是否存在矛盾或依赖未理清？
- 删除的旧逻辑是否确定不再有调用方依赖？

输出格式：
- ✅ 合理 + 从哪个维度确认的
- 🟡 存疑 + 具体场景 + 建议验证方式
- 🔴 不合理 + 哪个场景会出问题 + 建议方案

### 2.2 架构评审 / 2.3 存量适配影响

> **条件触发**：满足 SKILL.md「深度开关」条件时 → Read `section-architecture.md` 并按其 checklist 展开。
> 不满足时输出「**2.2/2.3 不适用**（本 PR 无架构重构或存量破坏性变更）」跳过。

### 2.4 测试方案梳理

> Read `section-testing.md` 并按其 checklist 展开。

2-3 段话概述：
- PR 的背景和目的
- 核心变更内容（按模块分组）
- 影响范围

## 四、代码审查（逐文件）

对每个变更文件审查：
- 代码正确性：逻辑错误、边界条件、空指针、类型安全
- 代码风格：命名规范、函数长度、注释质量
- 错误处理：遗漏处理、错误信息清晰度
- 可维护性：代码重复、魔法数字、硬编码
- 测试覆盖：关键路径是否有测试

输出格式：
- 🔴 **必须修改**：bug 或严重问题
- 🟡 **建议修改**：质量或可维护性
- 🟢 **可选优化**：锦上添花

每条包含：
```
<a id="issue-N"></a>
**🔴/🟡/🟢 I-N** — `<file_path>` L<line>
问题：<description>
建议：<suggestion / code example>
```

> ⚠️ 使用 `I-N` 编号（如 `I-1`），**不要**使用 `#N`。GitHub 会将 `#数字` 自动解析为 issue/PR 链接。

## 4.5、API / 配置变更清单

当 PR 涉及对外 API、配置、Schema 变更时列出（无则省略本章）：

| 类型 | 变更项 | 变更内容 | 向后兼容 | 备注 |
|------|--------|----------|----------|------|
| API | `endpoint` | 新增/删改 | ✅/⚠️/🔴 | 迁移指引 |
| 配置 | `config_key` | 默认值变更 | ✅ | 生效范围 |
| Schema | `ModelName` | 新增字段 | ✅ | 内部使用 |

检查范围：
- API 接口变更（HTTP/gRPC/WorkItem）
- 请求/响应字段增删改
- 配置项变更（YAML/环境变量/Feature Flag）
- 数据库/消息 schema 变更
- 依赖变更
