# 标准模式 — 完整审查 Checklist

## 报告结构

```
# Code Review: <PR title>

PR 基础信息
## 〇、总结（TL;DR）
## 一、PR 描述评审
## 二、方案评审（含测试方案梳理）
## 三、变更概述
## 四、代码审查（逐文件）
## 4.5、API / 配置变更清单
## 五、潜在风险检查
```

---

## PR 基础信息

```markdown
- **PR**: [#<number> <title>](<pr_url>)
- **分支**: `<head>` → `<base>` | **变更**: +<additions> / -<deletions>，<file_count> 个文件
- **作者**: <author(s)> | **关联 Issue**: #<number>（无则省略）
- **标签**: <labels>（无则省略）
```

---

## 〇、总结（TL;DR）

30 秒内让读者掌握全貌：
- 一句话概括 PR 做了什么
- 发现的问题统计（🔴 / 🟡 / 🟢 各多少条）
- 🔴 必须修改的问题摘要列表（来源不限于代码审查，涵盖所有章节）
- PR 描述质量：一句话评价
- 合并建议（建议合并 / 修复后合并 / 需要重大修改）

**🔴 问题来源说明**：必须修改的问题可来自任何章节——代码审查（bug/逻辑错误）、方案评审（设计不合理）、测试评审（UT/BVT 不达标、回归复现不充分）、风险检查（安全/并发问题）。只要达到 🔴 级别（不修就不能合并），都列入摘要。

---

## 一、PR 描述评审

对 PR body（及关联 Issue，如有）按四维结构评审：

**背景（Context）**
- 是否清晰描述当前现状和上下文？
- 读者能否仅凭描述理解问题所处的业务/技术场景？

**原因（Why）**
- 是否说明为什么要做？驱动力是什么？
- 问题的严重程度和影响范围是否有数据支撑？

**方案（How）**
- 是否描述了具体的技术方案或实现思路？
- 方案是否足够详细，能指导 Review？
- 是否讨论了备选方案及取舍理由？

**结果（Expected Outcome）**
- 是否明确预期结果和验收标准？
- 是否有可衡量的成功指标？

输出格式：
- ✅ 充分 + 简要说明
- 🟡 不足 + 缺什么 + 建议补充
- 🔴 缺失 + 为什么需要

---

## 二、方案评审

### 2.0 变更可视化（状态机 / 调用链 / 数据流图）

用图示说明本 PR 实际修改了什么。根据变更性质选择最合适的表达方式：

**选择依据**：
- 状态迁移/生命周期变更 → 状态机图
- 函数调用路径变更 → 调用链图
- 数据在模块间流转方式变更 → 数据流图
- 多种混合 → 选最能体现核心修改的一种，必要时组合

**输出格式**（ASCII 图 + 标注）：

```
修改前（简化）：
A → B → C → D
         ↑ 问题点

修改后：
A → B ──→ D     ← 移除 C 对 B 的依赖
     ↘ C（独立）  ← NEW / ← 修改
```

**标注规则**：
- `← NEW`：新增路径/节点
- `← 修改`：已有节点行为变更
- `← 移除`：删除的路径/节点（用删除线或注释）
- `← 问题点`：修复前的故障位置

**要求**：
- 图必须基于 diff 中实际代码变更绘制，不能是对 PR 描述的复述
- 涉及条件分支时标注分支条件
- 复杂场景可分"修改前"和"修改后"两张图对比

### 2.1 方案合理性

从三个维度综合评估：

**A. 函数调用场景**
- 变更函数在实际调用链中的位置和角色
- 调用方（caller）是否受到影响？是否需要感知此变更？
- 被调用方（callee）的契约是否被破坏？
- 是否有更简洁的实现方式（更少的调用层级、更直接的路径）？
- 是否有可复用的现有组件/库被忽略？

**B. 使用场景覆盖**
- 列出该变更涉及的所有使用场景（正常流、异常流、边界场景）
- 每个场景下方案是否正确处理？
- 是否存在未覆盖的场景会导致回归？
- 向后兼容性：现有调用方在不修改代码的情况下行为是否一致？

**C. 架构合理性**
- 职责划分是否清晰（变更是否放在正确的层/模块）？
- 抽象层次是否合理？是否过度设计或设计不足？
- 是否有更高效的算法或数据结构？

输出格式：
- ✅ 合理 + 从哪个维度确认的
- 🟡 存疑 + 具体场景 + 建议验证方式
- 🔴 不合理 + 哪个场景会出问题 + 建议方案

### 2.2 测试方案梳理（UT / BVT / Eval）

#### T1. 变更 → 测试映射

逐一检查每个新增/修改的**导出函数或核心内部函数**是否有对应 UT：

| 变更函数/方法 | 变更类型 | 对应 UT | 覆盖判定 |
|---------------|----------|---------|----------|
| `pkg.Func()` | 新增 | `test_func.py::test_xxx` | ✅ / 🟡 / ❌ |

#### T2. 场景覆盖矩阵

对已有 UT 检查以下维度：

| 场景 | 检查项 |
|------|--------|
| Happy path | 正常输入 → 预期输出 |
| Error path | 各类错误输入 → 正确错误处理 |
| 边界值 | 空字符串、超长输入、零值、最大值 |
| 空值/nil | nil 参数不 panic |
| 并发安全 | 多 goroutine 同时调用（如适用） |
| 幂等性 | 重复调用结果一致（如适用） |

#### T3. Mock 策略

- mock 层级是否正确？（应 mock 外部依赖，不 mock 内部实现）
- mock 返回值是否覆盖成功和失败？
- 是否存在"mock 了一切导致测试只验证了 mock 本身"？
- mock 行为是否与真实依赖一致？

#### T4. 测试质量红线

- 是否有有效断言？（禁止只调用不断言）
- 断言是否验证核心业务逻辑（非仅"不报错"）？
- 用例命名是否清晰表达测试意图？
- 是否使用 table-driven tests 覆盖多组输入？（Go 推荐）

#### T5. BVT 端到端场景覆盖

| 场景类型 | 检查项 |
|----------|--------|
| 核心正向流程 | 主要业务路径完整走通 |
| 关键分支流程 | 重要条件分支 |
| 错误恢复流程 | 异常输入/依赖故障时的降级 |
| 回归场景 | 历史 bug 的复现路径 |
| 并发场景 | 多请求并发时正确性 |

#### T6. 断言质量

- 是否精确验证业务结果（字段值、数据条数、状态码 + 响应体）？
- 是否过度宽松？常见：`status == 200` 不检查 body、`len >= 1` 代替精确值
- 断言失败的错误信息是否足够定位问题？

#### T7. 测试数据与环境管理

- 测试数据是否自包含？
- 测试间是否有数据隔离？
- setup/teardown 是否完整？
- 是否依赖特定执行顺序？
- 并发执行安全？

#### T8. 回归验证（Bug Fix PR 专项）

- 是否有专门的用例精确复现 bug 触发条件？
- 断言是否直接验证修复效果？
- 修复前失败、修复后通过？

#### T9. 故障复现效率（Bug Fix PR 专项）

评估测试用例作为"故障复现工具"的效率：

**复现速度**
- 测试能否在本地秒级/分钟级内复现故障？（理想 < 30s）
- 是否依赖外部服务/特定环境才能触发？（降低效率）
- 是否需要复杂的数据构造或前置步骤？
- CI 中单次运行耗时是否可接受？

**复现精度**
- 测试是否精确命中 bug 的触发条件（最小复现路径）？
- 还是通过大范围 E2E 间接覆盖（可能偶发通过/失败）？
- 如果 bug 回归，这个测试是否 100% 能捕获？

**复现隔离度**
- 测试失败时能否直接定位到具体函数/行？
- 还是需要进一步 debug 才能确认是这个 bug 复发？
- 错误信息是否包含足够上下文（输入值、期望值、实际值）？

**评级标准**：
- ⚡ 高效：本地秒级复现、精确命中、失败即定位
- 🟡 中等：需要特定环境或数据，但基本可靠
- 🐢 低效：依赖外部服务、复现慢、间接覆盖、需二次定位

输出示例：
```
⚡ 高效 — TestInfoExtractDSLStepOrder 直接断言 extract_fields 不含 page_selector，
  本地 <1s，回归即红，错误信息直接指向修复点。

🐢 低效 — TestGoWorker_DocumentParsePageSelectorRealPDFMinerU 依赖外部 MinerU 服务，
  超时 8min，CI 中 MinerU 不可用则 skip。作为 E2E 验证有价值，但不能作为唯一回归门禁。
```

#### Eval 方案（涉及 LLM/VLM 时）

- 变更是否影响 AI 输出质量？
- eval 数据集是否充分？
- eval 指标是否明确量化？
- 是否有与 baseline 的回归对比？

**测试补充建议格式**：

```
🧪 测试补充建议：

1. [UT] `pkg.Func()` — 缺少单元测试
   - 建议场景：a. 正常输入 b. 空输入 c. 超大输入
   - 建议 mock：mock XxxClient，分别返回成功/失败/超时
   - 优先级：🔴 必须 / 🟡 建议
```

输出格式：
- ✅ 方案合理 + 理由
- 💡 可优化建议 + 方案 + 预期收益
- 🧪 测试缺陷/缺失 + 补充策略

## 三、变更概述

2-3 段话概述：
- PR 的背景和目的
- 核心变更内容（按模块分组）
- 影响范围

## 四、代码审查（逐文件）

对每个变更文件审查：
- 代码正确性：逻辑错误、边界条件、空指针、类型安全
- 代码风格：命名规范、函数长度、注释质量
- 错误处理：遗漏处理、错误信息清晰度
- 可维护性：代码重复、魔法数字、硬编码
- 测试覆盖：关键路径是否有测试

输出格式：
- 🔴 **必须修改**：bug 或严重问题
- 🟡 **建议修改**：质量或可维护性
- 🟢 **可选优化**：锦上添花

每条包含：
```
<a id="issue-N"></a>
**🔴/🟡/🟢 #N** — `<file_path>` L<line>
问题：<description>
建议：<suggestion / code example>
```

## 4.5、API / 配置变更清单

当 PR 涉及对外 API、配置、Schema 变更时列出（无则省略本章）：

| 类型 | 变更项 | 变更内容 | 向后兼容 | 备注 |
|------|--------|----------|----------|------|
| API | `endpoint` | 新增/删改 | ✅/⚠️/🔴 | 迁移指引 |
| 配置 | `config_key` | 默认值变更 | ✅ | 生效范围 |
| Schema | `ModelName` | 新增字段 | ✅ | 内部使用 |

检查范围：
- API 接口变更（HTTP/gRPC/WorkItem）
- 请求/响应字段增删改
- 配置项变更（YAML/环境变量/Feature Flag）
- 数据库/消息 schema 变更
- 依赖变更

## 五、潜在风险检查

### 5.1 并发安全

- 共享资源竞态条件
- 锁粒度是否合理
- 死锁风险
- channel/goroutine 泄漏

### 5.2 性能

- N+1 查询
- 不必要的内存分配/拷贝
- 循环中可提取的计算
- 缺少索引的 DB 查询
- 慢查询风险

### 5.3 复杂度分析

- 关键函数时间复杂度（标注 Big-O）
- 空间复杂度可优化性
- 大数据量下性能退化风险
- 是否有更优复杂度替代

### 5.4 成本

- 不必要的 API 调用（云服务/第三方）
- 资源及时释放（连接/文件句柄）
- 缓存策略合理性
- 日志级别（生产避免过多 debug）

### 5.5 安全

- 输入校验
- SQL 注入 / XSS
- 敏感信息泄漏
- 权限检查完整性

### 5.6 LLM Prompt 质量（仅涉及 LLM/VLM 时）

**覆盖率**
- 不同语言/格式/异常输入是否有指令覆盖？
- 输出格式是否明确约束禁止项和边界行为？
- 是否有 negative examples（"不要做什么"）？

**稳定性**
- 输出解析是否有清洗/归一化（去代码块、JSON 包装等）？
- 是否有 fallback 策略应对 LLM 不遵守指令？
- 是否依赖精确格式（脆弱）vs 模糊匹配（健壮）？
- 多步调用中前步幻觉是否传播累积？

**可编程度**
- 能用正则/规则/代码做的事是否交给了 LLM？
- LLM 输出后是否有程序化校验层（列数/schema 验证等）？
- prompt 硬编码值是否应参数化？

**模型兼容**
- 同一 prompt 是否在多模型上运行？是否验证过效果？
- 是否有按模型分模板机制？
- 是否有模型特定指令，换模型会失效？

### 5.7 Token 效率（仅涉及 LLM/VLM 时）

**执行性能**
- 串行 vs 并行？是否有可并行化的串行调用？
- 总耗时与数据规模关系（N 页 = O(N) 延迟？）

**Token 复杂度**
- 输入 token：O(1) 固定 prompt vs O(N) 全量数据？
- 输出 token：期望输出与规模关系？
- 总 token = 调用次数 × 单次消耗（如 N × O(N) = O(N²)？）
- 是否有截断/摘要策略降 O(N) 为 O(1)？

**总体消耗**
- 与改动前相比总 token 增减？
- 是否有冗余信息重复发送？
- prompt 是否可精简/缓存/复用？

### 5.8 可插拔性

**可插拔**：核心策略是否以策略模式/接口解耦？切换模型/provider 成本？
**可开关**：新功能有 feature flag？关键参数可配置非硬编码？支持灰度？
**可替换**：逻辑职责单一边界清晰？接口为扩展留余地？
**可测试**：难以 mock 的外部依赖耦合？纯函数 vs 副作用比例？
**可调试**：中间状态日志？prompt/response 可追溯？A/B 对比方便？

### 5.9 违禁操作

- **超时硬编码**：超时值直接写死在业务代码中？
  - 必须通过配置项/环境变量/常量定义
  - 检查 HTTP client timeout、gRPC deadline、DB 查询超时、重试间隔
  - 测试代码中的可豁免但需标注

输出：🚫 违禁 + 位置 + 应改方式（一律🔴）

### 5.10 测试实现质量

**断言有效性**
- 已有测试是否因本次改动需更新？
- 断言是否验证核心逻辑（非仅副作用/日志）？
- 是否有"永远通过"的断言？

**测试代码 bug**
- expected 值是否正确？（复制粘贴忘改）
- 遗漏 cleanup / teardown？
- 测试中 goroutine 泄漏？
- 错误路径测试是否真触发了错误？

**Flaky 风险**
- timing 依赖（`time.Sleep`/固定超时）？
- ordering 依赖（执行顺序/map 遍历顺序）？
- 环境依赖（特定端口/路径/环境变量）？
- 并发测试竞态？

**可维护性**
- 过度耦合实现细节？
- 大量重复 setup 可提取为 helper？
- 失败错误信息足够定位？

输出：
- ⚠️ 风险 + 位置 + 修复建议
- 🧪 测试问题 + 位置 + 修复建议
- ✅ 已检查无问题的维度
