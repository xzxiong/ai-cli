## 前置：上下文探索

在执行 checklist 之前，先主动读取 diff 之外的关联代码，建立完整上下文：

### 探索步骤

1. **识别变更边界**
   - 列出本 PR 新增/删除/重命名的导出符号（函数、类型、常量）
   - 列出签名变更的函数

2. **搜索调用方**（diff 外）
   ```bash
   # 对每个被删除/重命名/签名变更的导出符号，搜索仓库内引用
   grep -rn "旧函数名\|旧类型名" --include="*.go" --include="*.ts" --include="*.tsx" | grep -v "_test.go"
   ```
   目的：确认是否有 PR 外的调用方未更新。

3. **追踪数据流入口**
   - 变更的核心函数从哪里被调用？（向上追 1-2 层 caller）
   - 输入数据从哪来？（API handler / WorkItem input / 配置文件）
   - 输出数据去哪？（下游消费者 / 持久化存储 / 前端展示）

4. **检查同层模块**
   - 同 package 下其他文件是否依赖被移动/删除的内部函数？
   - 是否有同层模块（如其他 WorkItem）使用相同的旧模式？（一致性）

5. **检查端到端路径**
   - 从用户触发入口（前端/API）到底层执行，走通一条完整路径
   - 标注路径上哪些节点被本 PR 修改了
   - 未修改的节点是否仍然兼容？

## 2.2 架构评审

### 判断原则

架构评审围绕 9 条原则，每条对应具体的检查切入点：

| # | 原则 | 核心问题 | 反例 | 检查切入点 |
|---|------|----------|------|-----------|
| 1 | **适配度** | 架构复杂度是否匹配问题复杂度？ | 简单问题过度抽象；复杂问题用 flat 结构硬塞 | 数文件数/层级数 vs 实际逻辑分支数 |
| 2 | **一致性** | 与仓库现有模式一致？偏离有充分理由？ | 同 package 其他 WorkItem 用模式 A，这个用 B | 对比同层模块结构 |
| 3 | **变更成本** | 下一个同类需求改动几个文件？ | 加一种 region kind 需改 5 处 switch | 模拟「假设新增 X，要改哪里」|
| 4 | **认知负载** | 新人能否从结构推断意图？ | 读 3 个文件才理解一个函数数据来源 | 检查文件命名、依赖层级深度 |
| 5 | **依赖方向** | 从不稳定指向稳定？无反向/环形？ | 工具包依赖业务逻辑 | 画 import 图，检查箭头方向 |
| 6 | **边界位置** | 模块边界在自然接缝上？ | 一个流程拆到两个模块、共享中间状态 | 看跨文件传递的参数数量 |
| 7 | **抽象封装** | 抽象是否隐藏了正确的细节？ | 调用方需知道内部实现才能正确使用；透传无收益 | 检查 public API 是否自解释、内部细节是否泄漏 |
| 8 | **可扩展性** | 新增同类功能能否低成本接入？ | 扩展需破坏现有接口或改 N 处 | 检查 switch/if-else 链是否可替换为注册/策略模式 |
| 9 | **可测试性** | 核心逻辑可否独立 UT？依赖可 mock？ | 纯逻辑耦合 IO 无法单测 | 检查函数参数是否依赖具体实现 vs 接口 |

### 如何使用

1. 对照上下文地图中的架构变更，逐条过 9 原则
2. 每条给出判定：✅ 满足 / 🟡 部分满足 + 改进建议 / 🔴 违反 + 风险说明
3. 不要求 9 条全列 — 只输出有发现的条目，无问题的原则可合并为一行「✅ 原则 1/2/4/5 无问题」

### 细化检查项

以下按关注领域分组，每个检查项背后对应 1-2 条原则：

#### A1. 文件/模块拆分（原则 1 适配度、4 认知负载、6 边界位置）

- 拆分后每个文件的职责能否用一句话描述？（认知负载）
- 拆分粒度是否匹配实际复杂度？（适配度）
- 文件命名是否准确反映内容？新读者能否从文件名推断职责？（认知负载）
- 循环依赖：拆分后是否引入文件间的循环 import / 调用环？（依赖方向）
- 一个完整逻辑流程是否被人为切断？跨文件传递的中间状态是否过多？（边界位置）
- 公共类型/工具函数的归属是否合理？是否有「垃圾桶文件」？（认知负载）

#### A2. 类型设计与接口契约（原则 7 抽象封装、8 可扩展性、9 可测试性）

- 新增类型是否有明确的不变量（invariant）？构造后是否始终有效？（抽象封装）
- 类型的可见性是否最小化？不需导出的不导出（抽象封装）
- 接口是否内聚？是否有「大而全」的接口应拆为小接口？（抽象封装）
- 新增类型/功能时是否需要修改现有类型定义？（可扩展性）
- 类型之间的关系是否清晰？组合 vs 嵌入是否选对？（认知负载）
- 旧类型的字段删除/重命名：JSON 反序列化是否安全？（边界位置）
- 核心逻辑的输入是否为接口/值类型，而非具体 struct 或外部 client？（可测试性）

#### A3. 模块间耦合与依赖（原则 2 一致性、3 变更成本、5 依赖方向）

- 新增的模块间依赖关系：方向是否从上层指向下层？（依赖方向）
- 是否存在「绕路调用」（A 调 B 只为拿 C，应直接依赖 C）？（变更成本）
- 是否存在内部实现细节泄漏到外部模块？（抽象封装）
- WorkItem/Handler 之间是否存在隐式调用（直接 new 内部组件调 Handle）？（可测试性、一致性）
- 与同层模块相比，依赖模式是否一致？（一致性）
- 如果新增一个类似模块，能否复用现有依赖结构？（可扩展性）

### 输出格式

```
#### 架构判定

✅ 原则 1/4/5/6 无问题
🟡 原则 3（变更成本）— 新增 region kind 需改 engineeringRegionKindIsValid + engineeringRegionIsTableLike 两处，建议改为注册表
🔴 原则 9（可测试性）— expandDocumentVisualTextIndexDocuments 内部直接 new MultiLevelIndex 调 Handle，无法 mock

每条 🟡/🔴 附带：
- 具体位置（文件:行号）
- 影响说明
- 改进建议
```

## 2.3 存量适配影响

系统性检查本次变更对存量系统的影响：

### B1. 存量数据兼容

| 数据类型 | 变更内容 | 旧数据反序列化 | 兼容策略 | 判定 |
|----------|----------|----------------|----------|------|
| 例：manifest JSON | 新增 table_summary 字段 | omitempty 零值 | 自动兼容 | ✅ |
| 例：object 结构 | 删除 RetryCount 字段 | 旧 JSON 含此字段 | Go 自动忽略 | ✅ |

检查清单：
- 新增字段：是否有 omitempty / default 保证旧数据可读？
- 删除字段：旧数据中含此字段时是否安全忽略？下游是否依赖？
- 重命名字段：是否有旧名 → 新名的兼容层？还是直接断裂？
- 结构体嵌套层级变更：深路径访问（`a.b.c`）是否断裂？

### B2. 存量调用方影响

> 基于前置探索步骤 2「搜索调用方」的结果填写此表。

| 调用方 | 依赖的接口/类型 | 本次变更 | 是否需要改动调用方 | 判定 |
|--------|----------------|----------|-------------------|------|
| 例：explore agent | image_vector_table 变量 | 改从 semantic_model.files 读取 | 需确认配置源 | 🟡 |

检查清单：
- 函数签名变更：调用方是否已同步更新？（grep 确认，不能仅靠 diff）
- 删除的导出函数/类型：是否有同仓库或跨仓库调用方？
- 行为变更（同签名不同语义）：调用方是否依赖旧行为？
- 错误处理变更：新增的 error return 是否被调用方正确处理？
- 跨仓库依赖：如果是 SDK/共享包，其他仓库是否 vendor 了旧版本？

### B3. 存量配置/模板兼容

| 配置项 | 变更内容 | 旧配置是否仍有效 | 降级行为 | 判定 |
|--------|----------|-----------------|----------|------|
| 例：form field_id | vector_index → drawing_index | 旧模板引用旧名 | 前端找不到字段 | ⚠️ |

检查清单：
- DSL/模板变量名变更：已持久化的 workflow 是否会断裂？
- 配置项删除：有无 graceful fallback？
- 默认值变更：现有未显式设置的部署是否受影响？
- Feature flag：新行为是否有开关保护？

### B4. 迁移路径（如有破坏性变更）

- 是否需要数据迁移脚本？
- 是否需要灰度/分批上线？
- 回滚方案是什么？回滚后旧数据是否完整？
- 是否有文档/changelog 告知使用者？

输出格式：
- ✅ 兼容 + 机制说明
- 🟡 需确认 + 什么条件下可能出问题
- 🔴 不兼容 + 影响范围 + 建议修复方式
