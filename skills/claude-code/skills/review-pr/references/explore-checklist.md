# 探索模式 — 审查 Checklist

面向 Demo / PoC / 探索性方案的 PR。目标是验证可行性而非生产就绪。

## 报告结构

```
# Code Review: <PR title>

PR 基础信息
## 〇、总结（TL;DR）
## 一、方案全景梳理
## 二、逻辑自洽性检查
## 三、可观测性评估
## 四、可迭代性评估
## 五、阻塞性问题
```

---

## PR 基础信息

```markdown
- **PR**: [#<number> <title>](<pr_url>)
- **分支**: `<head>` → `<base>` | **变更**: +<additions> / -<deletions>，<file_count> 个文件
- **作者**: <author(s)> | **关联 Issue**: #<number>（无则省略）
- **模式**: 🧪 探索性 Review
```

---

## 〇、总结（TL;DR）

- 一句话概括：这个 PR 在探索什么方案
- 方案成熟度：🟢 可直接迭代 / 🟡 需补充后可迭代 / 🔴 方案需重新设计
- 关键发现摘要（每条一行）
- 下一步建议（最多 3 条 action item）

---

## 一、方案全景梳理

**这是探索模式最核心的章节**。帮助读者建立对方案的完整认知。

### 1.1 方案目标与边界

- 解决什么问题？
- 适用范围和前提假设？
- 本次要验证什么？明确不做什么？

### 1.2 数据流图

梳理完整数据流，从输入到输出的每步变换：

```
输入(file_data) → [阶段A] → 中间产物X → [阶段B] → 中间产物Y → ... → 最终输出
```

每个阶段标注：
- 输入数据的格式和来源
- 输出数据的格式和去向
- 数据量变化趋势（1:1 / 1:N 扇出 / N:1 聚合）
- 是否有数据丢失风险（输入被静默丢弃）

### 1.3 第三方组件交互

| 组件 | 交互方式 | 调用频次 | 失败影响 | 容错策略 |
|------|----------|----------|----------|----------|
| VLM 模型 | HTTP API | N页×(1+T)次 | 单页降级 | 超时+重试 |
| S3 | SDK | 每图1次 | 图片丢失 | 日志告警 |
| PaddleOCR | 本地 | 每图1次 | OCR缺失 | 跳过 |

重点关注：
- 调用频次与数据规模关系（O(1)/O(N)/O(N²)）
- 是否有连接池/客户端复用？
- 失败降级策略是否合理？
- 不必要的强依赖能否改为弱依赖？

### 1.4 模块拆解

- 按模块/阶段梳理职责和交互
- 标注每模块的输入/输出/依赖
- 识别关键路径和风险点

输出格式：
- 📐 方案结构清晰的部分
- ❓ 方案模糊或缺失 + 需补充什么
- 💡 可优化方向（标注"迭代建议"，不要求立即修改）

---

## 二、逻辑自洽性检查

### 2.1 描述 vs 实现

- PR 描述的方案与代码是否一致？
- 有无描述了但未实现的部分？（"TODO" vs "遗漏"）
- 有无实现了但描述未提及的部分？

### 2.2 模块间一致性

- 接口约定是否匹配（输入输出格式、错误协议）？
- 数据流是否完整，有无断裂？
- 配置/开关语义是否各处一致？

### 2.3 假设与约束

- 前提假设是否成立？（如"模型输出一定是 JSON"）
- 隐含的顺序/状态依赖是否显式声明？
- 边界条件处理是否与描述一致？

### 2.4 并发模型

- 哪些阶段并发？粒度？（文档级/页面级/元素级）
- 并发度由什么控制？可配置？硬编码？
- 多层并发叠加放大？（外 M × 内 N = M×N 实际并发）
- 下游资源能否承受最大并发？全局限流？
- 超时和取消机制完备？
- 部分失败的处理策略？结果顺序保证？共享状态竞态？

输出格式：
- ✅ 逻辑一致
- ⚠️ 不一致/矛盾 + 位置 + 如何对齐
- ❓ 无法从代码判断 + 需作者确认

---

## 三、可观测性评估

### 3.1 效果数据

- 有量化效果指标？（准确率/召回率/耗时/成本）
- 指标有采集和输出机制？
- 有 baseline 对比？

### 3.1.1 LLM/VLM 量化指标（仅涉及大模型时）

**质量指标**

| 指标 | 含义 | 采集方式 |
|------|------|----------|
| 元素检出率/字段召回率 | 覆盖多少目标内容 | eval + ground truth |
| 类型/分类准确率 | 判断正确比例 | eval |
| 内容准确率 | 与 ground truth 相似度 | eval |
| 格式合规率 | 符合预期 schema 比例 | 运行时统计 |
| 回归对比 | 与改动前差异 | A/B eval |

**效率指标**

| 指标 | 含义 | 采集方式 |
|------|------|----------|
| 单次调用耗时 P50/P95 | 延迟分布 | Prometheus Histogram |
| 端到端耗时 | 完整链路总时间 | Prometheus Histogram |
| Token 消耗 | input + output tokens | API 响应 |
| 调用次数 | 每任务 LLM 调用数 | Counter |
| 单位成本 | 每任务推理成本 | token × 单价 |

**稳定性指标**

| 指标 | 含义 | 采集方式 |
|------|------|----------|
| 任务成功率 | 成功占比 | Counter |
| 降级率 | 触发 fallback 比例 | Counter |
| 格式解析失败率 | 无法解析为目标 schema | Counter |
| 超时率 | 调用超时比例 | Counter |

审查要点：上述指标是否已定义？已埋点？

### 3.2 过程数据

- 关键中间步骤有日志输出？
- 能追溯单次请求完整链路？
- Debug 模式有更详细输出？

### 3.3 异常数据

- 失败/降级/超时有指标和日志？
- 能区分不同失败原因？
- 告警阈值合理？

输出格式：
- ✅ 可观测性充分
- 📊 缺少的数据输出 + 建议补充 + 为什么需要
- 💡 可增强观测点（非必须）

---

## 四、可迭代性评估

### 4.1 可开关

- 有总开关？能一键关闭回退？
- 关键参数可配置？
- 支持灰度？

### 4.2 可替换

- 核心组件可独立替换？
- 替换时需改主流程？
- 接口为未来扩展留余地？

### 4.3 可测试

- 核心逻辑易于单测？（纯函数 vs 副作用）
- 外部依赖可 mock？
- 有 eval 框架支撑回归？

### 4.4 可清理

- 方案被否决时清理成本？
- 有侵入性修改影响现有代码？
- 探索代码与主干边界清晰？

输出格式：
- ✅ 迭代友好
- 🔧 迭代障碍 + 位置 + 改进建议
- 💡 可增强扩展点

---

## 五、阻塞性问题

探索模式下不做全面代码审查，**仅**检查：

- 🔴 crash / panic / 未捕获异常
- 🔴 数据丢失或损坏（静默吞错、写错位置）
- 🔴 敏感信息泄漏（硬编码密钥、内网地址暴露）
- 🔴 资源泄漏（goroutine/线程/连接/文件句柄未释放）

无阻塞性问题时输出：✅ 未发现阻塞性问题

---

## 探索模式审查原则

1. **方案优先**：重点是"方案对不对"，非"代码好不好"
2. **数据驱动**：关注方案能否产出可评估的数据
3. **迭代友好**：关注是否为后续演进留路
4. **宽容细节**：不纠结风格、命名、微小性能
5. **标注 TODO**：当前可接受但生产化必须解决 → `📌 生产化 TODO`
