# 项目共识/目标：除幻 · 省 Token · 快速迭代

> 适用范围: 所有涉及 LLM/VLM 调用链的 PR Review 和代码变更
> 使用方式: 在 Kiro 聊天中通过 `#project-goals` 引用

本文件定义三大项目目标的审查维度和输出规范。仅在 PR 涉及相关领域时触发对应审查。

---

## 一、除幻（LLM/VLM Prompt 质量）

> 触发条件: PR 涉及 LLM/VLM prompt 或 AI 调用链

### 1.1 Prompt 覆盖率

审查 prompt 是否覆盖了所有关键场景和边界条件：

- 输入变体: 不同语言、不同格式、异常输入是否都有指令覆盖
- 输出约束: 是否明确约束了输出格式、禁止项、边界行为
- 负面示例: 是否提供了"不要做什么"的 few-shot negative examples

### 1.2 Prompt 稳定性

评估对 LLM 输出随机性的容错能力：

- 输出解析是否有清洗/归一化逻辑（去代码块标记、JSON 包装等）
- 是否有 fallback 策略应对 LLM 不遵守指令
- 是否依赖精确格式（脆弱）vs 模糊匹配（健壮）
- 多步调用中前一步幻觉是否会传播累积

### 1.3 可编程程度

评估 prompt 中有多少逻辑可以用确定性代码替代：

- 能用正则/规则/代码做的事不要交给 LLM（格式校验、数值计算等）
- LLM 输出后是否有程序化校验层（列数校验、schema 验证等）
- prompt 中的硬编码值是否应该参数化

### 输出格式

```
🧠 幻觉风险 + 具体 prompt 位置 + 改进建议
✅ 已检查无问题的维度
```

## 二、省 Token & 性能（LLM/VLM 调用效率 + 传统延迟/吞吐）

> 触发条件: PR 涉及 LLM/VLM 调用，或涉及数据处理管线、外部服务调用、批量操作

### 2.1 Token 复杂度

分析 token 消耗与数据规模 N 的关系：

- 输入 token: O(1) 固定 prompt vs O(N) 全量数据？
- 输出 token: 期望输出与数据规模的关系
- 总 token = 调用次数 × 单次消耗（如 N 次 × O(N) = O(N²)）
- 是否有截断/摘要策略将 O(N) 降为 O(1)

### 2.2 Token 总体消耗

- 与改动前相比总 token 增减？
- 是否有冗余信息重复发送（如每次都发完整上下文）
- prompt 模板是否可精简？是否可缓存/复用减少调用？

### 2.3 延迟 & 执行性能

评估端到端延迟和吞吐量：

- 串行 vs 并行: 是否有可并行化的串行调用？（LLM 调用、外部服务请求、文件 IO）
- 总耗时与数据规模的关系（如 N 页 = O(N) 延迟 vs 并行后 O(N/P)）
- 关键路径分析: 哪一步是延迟瓶颈？LLM 调用 / 网络 IO / 文件处理 / DB 查询？
- 是否有不必要的同步等待（如等待非关键路径完成才继续）

### 2.4 网络 & IO 效率

- HTTP 调用: 是否复用连接（connection pooling）？是否有不必要的重复请求？
- 文件 IO: 大文件是否流式处理？是否有全量加载到内存的风险？
- 数据库: 是否有 N+1 查询？批量操作是否用了 batch insert/update？
- 对象存储 (MinIO): 上传/下载是否用了 multipart？大量小文件是否可合并？

### 2.5 资源占用

- 内存: 处理大文件/大批量时的峰值内存是否可控？是否有流式替代方案？
- Goroutine/协程: 并发数是否有上限控制？是否可能因无限并发打爆下游服务？
- 超时: 长链路调用（LLM → 解析 → 存储）的超时是否合理？是否有级联超时设计？

### 2.6 缓存 & 去重

- 相同输入是否会重复调用 LLM？是否有结果缓存机制？
- 幂等性: 重试/重跑时是否会产生重复计算或重复写入？
- 中间结果: 多步管线中间结果是否持久化？失败重跑是否需要从头开始？

### 输出格式

```
💰 Token 效率问题 + 具体位置 + 优化建议 + 预估节省
⏱️ 延迟/性能问题 + 具体位置 + 优化建议 + 预估改善
✅ 已检查无问题的维度
```

## 三、快速迭代（可插拔性与演进能力）

> 触发条件: 所有变更均需评估

### 3.1 可插拔

核心策略/算法是否以可插拔方式实现：

- 是否通过策略模式、接口抽象等方式解耦？能否不改主流程就替换实现？
- LLM/VLM 调用是否有清晰的接口层？切换模型/provider 的成本如何？
- 新增的处理步骤能否独立启用/禁用？

### 3.2 可开关

关键行为是否支持配置化控制：

- 新功能是否有 feature flag 或配置项控制启停？
- 关键参数（截断行数、重试次数、阈值、模型选择等）是否可配置而非硬编码？
- 是否支持灰度（按租户/文件类型等维度逐步开启）？

### 3.3 可替换

当前实现能否低成本替换为更优方案：

- 核心逻辑是否职责单一、边界清晰？能否独立演进不影响其他模块？
- 数据结构和接口设计是否为未来扩展留有余地？

### 3.4 可测试

新增逻辑是否易于单元测试：

- 是否有难以 mock 的外部依赖耦合？
- 纯函数 vs 有副作用的方法比例是否合理？

### 3.5 可调试

出问题时能否快速定位：

- 是否有足够的中间状态日志？prompt 和 LLM 响应是否可追溯？
- 新旧方案能否方便地 A/B 对比？

### 输出格式

```
🔧 迭代障碍 + 具体位置 + 改进建议
✅ 已检查无问题的维度
```
