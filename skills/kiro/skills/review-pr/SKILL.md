# Review PR Skill

## 目的
对 PR 进行全面 Code Review，输出结构化的审查报告。

## Review 模式

### 标准模式（默认）
面向在稳定系统上添加成熟方案的 PR。全面审查代码质量、性能、安全、并发等。

### 探索模式（`--explore`）
面向 Demo / PoC / 探索性方案的 PR。这类 PR 的目标是验证可行性而非生产就绪，审查侧重点不同：

| 维度 | 标准模式 | 探索模式 |
|------|----------|----------|
| 方案梳理 | 评估是否有更优方案 | **重点**：完整梳理方案全貌，评估技术路线合理性 |
| 逻辑自洽 | 检查代码正确性 | **重点**：方案描述与实现是否一致，各模块间逻辑是否自洽 |
| 可观测性 | 检查日志/指标 | **重点**：是否有足够的数据输出支撑方案评估（指标、日志、debug 工具） |
| 可迭代性 | 检查可插拔/可开关 | **重点**：方案能否低成本演进，是否为后续迭代留好了扩展点 |
| 代码质量 | 逐文件详细审查 | **降低**：只关注阻塞性问题（crash/数据丢失），不纠结风格和细节 |
| 性能/并发 | 详细分析 | **降低**：只标注明显的资源泄漏或死锁风险，不做深度优化建议 |
| 安全 | 详细检查 | **降低**：只检查敏感信息泄漏（硬编码密钥等） |

**模式判断**：
1. 用户显式指定 `--explore` → 探索模式
2. 用户说"探索性review"、"demo review"、"poc review" → 探索模式
3. 否则 → 标准模式

## 使用方法
```bash
# 标准模式
kiro chat "review pr https://github.com/matrixorigin/matrixflow/pull/<PR_NUMBER>"
kiro chat "review pr #<PR_NUMBER>"

# 探索模式
kiro chat "review pr #<PR_NUMBER> --explore"
kiro chat "探索性review #<PR_NUMBER>"

# 可选：关联 Issue 一起评审
kiro chat "review pr #<PR_NUMBER> --with-issue"
```

## Skill 逻辑

### ⚠️ 重要：执行模式

`review-pr-agent` 没有 bash 执行权限。主代理必须：
1. 自己执行所有 `gh` CLI 命令获取 PR 数据
2. 将获取到的数据（元数据 + diff）作为 `relevant_context` 传递给 `review-pr-agent`
3. 让 `review-pr-agent` 仅负责分析和生成报告

### 1. 获取 PR 元数据（主代理执行）
```bash
gh pr view <PR_NUMBER> --json title,body,files,commits,additions,deletions,baseRefName,headRefName,labels,state
```

### 2. 获取代码 diff（主代理执行）
```bash
# 文件列表及 diff 位置索引
gh pr diff <PR_NUMBER> 2>&1 | grep -n '^diff --git'

# 按需读取核心代码变更（跳过生成文件如 swagger/docs）
gh pr diff <PR_NUMBER> 2>&1 | sed -n '<START>,<END>p'
```

注意：
- 跳过自动生成文件（swagger.json、docs.go 等）
- 大 PR 先定位文件边界，按需读取关键文件 diff
- 优先读取业务逻辑代码（handler、service、model、repo）

### 3. 获取关联 Issue（可选，需用户明确要求）
仅当用户在发起 review 时指定了 `--with-issue` 或明确要求"关联 Issue 一起看"时才执行。默认不获取。

从 PR body 或 commit message 中提取 issue 编号：
```bash
gh issue view <ISSUE_NUMBER> --json title,body
gh api repos/<OWNER>/<REPO>/issues/<ISSUE_NUMBER>/comments --jq '.[].body'
```

### 4. 委托 review-pr-agent 生成审查报告

将步骤 1-3 收集的所有数据作为 `relevant_context` 传递给 `review-pr-agent`，让其生成审查报告。

**传递 review 模式**：在 `relevant_context` 中明确标注 `REVIEW_MODE: standard` 或 `REVIEW_MODE: explore`，让 agent 选择对应的报告模板。

按以下维度输出结构化报告（根据 review 模式选择对应模板）：

#### 〇、总结（TL;DR）

- 一句话概括：这个 PR 在探索什么方案
- 方案成熟度评估：🟢 可直接迭代 / 🟡 需补充后可迭代 / 🔴 方案需重新设计
- 关键发现摘要（每条一行）
- 下一步建议（最多 3 条最重要的 action item）

#### 一、方案全景梳理

**这是探索模式最核心的章节**。目标是帮助读者（包括方案作者自己）建立对方案的完整认知。

**1.1 方案目标与边界**
- 这个方案要解决什么问题？
- 方案的适用范围和前提假设是什么？
- 哪些是本次探索要验证的，哪些是明确不做的？

**1.2 数据流图**

梳理方案的完整数据流，从输入到输出的每一步数据变换。用文本流程图或表格呈现：

```
输入(file_data) → [阶段A] → 中间产物X → [阶段B] → 中间产物Y → ... → 最终输出
```

对每个阶段标注：
- 输入数据的格式和来源
- 输出数据的格式和去向
- 数据量的变化趋势（1:1 / 1:N 扇出 / N:1 聚合）
- 是否有数据丢失风险（某些输入在流程中被静默丢弃）

**1.3 第三方组件交互**

列出方案依赖的所有外部组件/服务，逐个评估交互方式：

| 组件 | 交互方式 | 调用频次 | 失败影响 | 容错策略 |
|------|----------|----------|----------|----------|
| 如：VLM 模型服务 | HTTP API | N 页 × (1+1+T) 次 | 单页降级 | 超时+重试 |
| 如：S3 | SDK | 每张图片 1 次 | 图片丢失 | 日志告警 |
| 如：PaddleOCR | 本地调用 | 每张图片 1 次 | OCR 缺失 | 跳过 |

重点关注：
- 调用频次与数据规模的关系（O(1) / O(N) / O(N²)）
- 是否有连接池/客户端复用，还是每次新建？
- 失败时的降级策略是否合理（重试 / 跳过 / 整体失败）
- 是否有不必要的强依赖可以改为弱依赖？

**1.4 模块拆解**
- 按模块/阶段梳理各部分的职责和交互关系
- 标注每个模块的输入/输出/依赖
- 识别方案中的关键路径和风险点

输出格式：
- 📐 方案结构清晰的部分
- ❓ 方案描述模糊或缺失的部分 + 需要补充什么
- 💡 方案可优化的方向（不要求立即修改，标注为"迭代建议"）

#### 二、逻辑自洽性检查

检查方案从描述到实现是否逻辑一致，不自相矛盾。

**2.1 描述 vs 实现**
- PR 描述/Issue 中声称的方案，与代码实现是否一致？
- 是否有描述了但未实现的部分？（标注为"TODO"还是"遗漏"）
- 是否有实现了但未在描述中提及的部分？

**2.2 模块间一致性**
- 各模块的接口约定是否匹配（输入输出格式、错误处理协议）？
- 数据流是否完整，有无断裂（A 模块输出的数据 B 模块是否能消费）？
- 配置项/开关的语义是否在各处一致？

**2.3 假设与约束**
- 方案依赖的前提假设是否成立？（如"模型输出一定是 JSON"）
- 是否有隐含的顺序依赖或状态依赖未显式声明？
- 边界条件处理是否与方案描述一致？（空输入、超大输入、异常输入）

**2.4 并发模型**
- 方案中是否存在并发处理？如果有，梳理完整的并发拓扑：
  - 哪些阶段是并发的？并发粒度是什么？（文档级 / 页面级 / 元素级）
  - 并发度由什么控制？是否可配置？是否有硬编码的并发数？
  - 多层并发是否存在叠加放大？（如外层 M 并发 × 内层 N 并发 = M×N 实际并发）
- 并发对下游资源的压力是否可控？
  - 下游服务（模型服务、数据库、S3）能否承受最大并发量？
  - 是否有全局限流机制（Semaphore / 令牌桶 / 共享线程池）？
  - 超时和取消机制是否完备？（单任务超时、整体超时、context 取消传播）
- 并发结果的聚合是否正确？
  - 部分失败时的处理策略是什么？（跳过 / 重试 / 整体失败）
  - 结果顺序是否有保证？（如页面顺序）
  - 共享状态是否有竞态风险？

输出格式：
- ✅ 逻辑一致的部分
- ⚠️ 不一致或矛盾的地方 + 具体位置 + 建议如何对齐
- ❓ 无法从代码判断的假设 + 需要作者确认

#### 三、可观测性评估

探索性方案的核心价值在于产出数据来验证方案可行性。如果方案跑完了但没有数据输出，就无法评估效果。

**3.1 效果数据**
- 方案是否有量化的效果指标？（准确率、召回率、耗时、成本等）
- 这些指标是否有采集和输出机制？（Prometheus、日志、eval 报告）
- 是否有 baseline 对比？（与改动前/竞品方案的对比数据）

**3.1.1 LLM/VLM 量化指标** — 仅当 PR 涉及 LLM/VLM 调用时展开

当探索方案的核心是大模型调用时，必须有以下三维量化指标才能评估方案价值：

**质量指标（效果好不好）**

| 指标 | 含义 | 采集方式 |
|------|------|----------|
| 元素检出率 / 字段召回率 | 模型输出覆盖了多少目标内容 | eval 数据集 + ground truth |
| 类型/分类准确率 | 模型判断正确的比例 | eval 数据集 |
| 内容准确率 | 输出文本与 ground truth 的相似度（edit distance / BLEU / 精确匹配） | eval 数据集 |
| 格式合规率 | 输出符合预期 schema 的比例（JSON 解析成功率、字段完整率） | 运行时统计 |
| 回归对比 | 与改动前方案在同一数据集上的质量差异 | A/B eval |

审查要点：是否有 eval 数据集？是否有 ground truth？是否有自动化的质量评估脚本？

**效率指标（快不快、贵不贵）**

| 指标 | 含义 | 采集方式 |
|------|------|----------|
| 单次调用耗时 P50/P95 | 单次 LLM/VLM 请求的延迟分布 | Prometheus Histogram |
| 端到端耗时 | 完整处理链路的总时间 | Prometheus Histogram |
| Token 消耗 | 每次调用的 input_tokens + output_tokens | API 响应中提取 |
| 单任务总 Token | 一个任务的总 token（= 调用次数 × 单次消耗） | 聚合统计 |
| 调用次数 | 每个任务触发的 LLM/VLM 调用次数 | Counter |
| 单位成本 | 每任务/每页/每文档的推理成本 | token × 单价 |

审查要点：API 响应中的 usage 字段是否被采集？是否有 token 消耗的 Prometheus 指标或日志？

**稳定性指标（稳不稳）**

| 指标 | 含义 | 采集方式 |
|------|------|----------|
| 任务成功率 | 成功完成的任务占比 | Counter |
| 降级率 | 触发 fallback 的比例 | Counter |
| 格式解析失败率 | 模型输出无法解析为目标 schema 的比例 | Counter |
| 兜底机制触发率 | 补偿/重试/兜底逻辑被触发的频率 | Counter |
| 超时率 | LLM/VLM 调用超时的比例 | Counter |

审查要点：上述指标是否已定义？是否已在代码中埋点？是否有未定义或定义了但未埋点的指标？

**3.2 过程数据**
- 关键中间步骤是否有日志输出？（输入、输出、耗时、错误）
- 是否能追溯单次请求的完整处理链路？
- Debug 模式下是否有更详细的数据输出？（如 bbox 可视化、prompt/response 记录）

**3.3 异常数据**
- 失败/降级/超时等异常路径是否有指标和日志？
- 是否能区分不同类型的失败原因？
- 告警阈值是否合理？

输出格式：
- ✅ 可观测性充分的部分
- 📊 缺少的数据输出 + 建议补充什么指标/日志 + 为什么需要
- 💡 可增强的观测点（非必须，但有助于方案评估）

#### 四、可迭代性评估

探索性方案最终要演进为生产方案。评估当前实现是否为后续迭代留好了路。

**4.1 可开关**
- 方案是否有总开关？能否一键关闭回退到旧方案？
- 关键参数是否可配置？（模型选择、并发数、超时、阈值等）
- 是否支持灰度？（按租户/文件类型/比例逐步开启）

**4.2 可替换**
- 核心组件是否可独立替换？（如换模型、换 prompt、换处理策略）
- 替换时是否需要改动主流程代码？
- 接口设计是否为未来扩展留有余地？

**4.3 可测试**
- 核心逻辑是否易于单元测试？（纯函数 vs 有副作用）
- 外部依赖是否可 mock？
- 是否有 eval 框架支撑效果回归测试？

**4.4 可清理**
- 如果方案最终被否决，清理成本有多大？
- 是否有侵入性修改影响了现有代码？
- 探索代码与主干代码的边界是否清晰？

输出格式：
- ✅ 迭代友好的设计
- 🔧 迭代障碍 + 具体位置 + 改进建议
- 💡 可增强的扩展点

#### 五、阻塞性问题（仅限 crash / 数据丢失 / 安全泄漏）

探索模式下不做全面代码审查，只检查会导致以下后果的问题：
- 🔴 程序 crash / panic / 未捕获异常
- 🔴 数据丢失或损坏（静默吞错、写入错误位置）
- 🔴 敏感信息泄漏（硬编码密钥、内网地址暴露）
- 🔴 资源泄漏（goroutine/线程/连接/文件句柄未释放）

如果没有阻塞性问题，输出：✅ 未发现阻塞性问题

---

### 探索模式审查原则

1. **方案优先**：重点是"方案对不对"，而非"代码好不好"
2. **数据驱动**：关注方案是否能产出可评估的数据
3. **迭代友好**：关注方案是否为后续演进留好了路
4. **宽容细节**：不纠结代码风格、命名规范、微小性能问题
5. **标注 TODO**：对于"当前可以接受但生产化时必须解决"的问题，标注为 `📌 生产化 TODO` 而非 🔴

#### 〇、总结（TL;DR）

放在报告最前面，让读者 30 秒内掌握全貌：
- 一句话概括 PR 做了什么
- 发现的问题统计（🔴 / 🟡 / 🟢 各多少条）
- 🔴 必须修改的问题摘要列表（每条一行）
- PR 描述质量：一句话总结描述是否完整，哪些维度缺失
- 合并建议（建议合并 / 修复后合并 / 需要重大修改）

#### 一、PR 描述评审（背景 / 原因 / 方案 / 结果）

对 PR body（以及关联 Issue body，如有）进行描述质量评审。PR 描述是 Reviewer 理解变更的第一入口，质量直接影响 Review 效率和准确性。

> 评审范围：默认仅评审 PR body。若用户要求关联 Issue，则将 Issue body 及其所有 comments 一并纳入评审（comments 中常包含方案讨论、技术决策等关键信息）。

**背景（Context）**
- 是否清晰描述了当前现状和上下文？
- 读者能否仅凭描述理解问题所处的业务/技术场景？

**原因（Why）**
- 是否说明了为什么要做这件事？驱动力是什么（用户反馈、性能瓶颈、技术债、新需求）？
- 问题的严重程度和影响范围是否有数据支撑？

**方案（How）**
- 是否描述了具体的技术方案或实现思路？
- 方案是否足够详细，能指导 Review（而非仅一句话概括）？
- 是否讨论了备选方案及取舍理由？

**结果（Expected Outcome）**
- 是否明确了预期结果和验收标准？
- 是否有可衡量的成功指标（性能提升 X%、错误率降至 Y 等）？

输出格式：
- ✅ 描述充分的维度 + 简要说明
- 🟡 描述不足的维度 + 缺少什么 + 建议补充的内容
- 🔴 完全缺失的维度 + 为什么需要补充

#### 二、变更概述（PR 做了什么）

用 2-3 段话概述：
- 这个 PR 的背景和目的
- 核心变更内容（按模块分组）
- 影响范围

#### 三、方案评审（是否有更优方案）

评估当前实现方案，考虑：
- 是否有更简洁的实现方式
- 是否有更高效的算法或数据结构
- 是否有可复用的现有组件/库被忽略
- 架构设计是否合理（职责划分、抽象层次）
- 是否过度设计或设计不足

**测试方案梳理**（UT / BVT / Eval）

对 PR 中的测试变更进行方案级评审，不仅看"有没有测试"，更要看"测试方案是否合理、场景是否覆盖完整"。

**UT（单元测试）方案**

*3.T1 变更 → 测试映射检查*

逐一检查 PR 中每个新增/修改的**导出函数或核心内部函数**，确认是否有对应的 UT 覆盖：

| 变更函数/方法 | 变更类型 | 对应 UT | 覆盖判定 |
|---------------|----------|---------|----------|
| `pkg/handler.ExtractText()` | 新增 | `handler_test.go:TestExtractText` | ✅ 已覆盖 |
| `pkg/service.ParseLayout()` | 修改签名 | 无 | ❌ 缺失 |
| `internal/util.CleanJSON()` | 新增 | `util_test.go:TestCleanJSON` | 🟡 仅 happy path |

对每个 ❌ 或 🟡 的条目，给出具体补充建议（见下方输出格式）。

*3.T2 场景覆盖矩阵*

对 PR 中已有的每个 UT，检查是否覆盖了以下场景维度：

| 场景维度 | 检查项 | 示例 |
|----------|--------|------|
| Happy path | 正常输入 → 预期输出 | 标准 JSON 输入返回正确结构 |
| Error path | 各类错误输入 → 正确的错误处理 | 无效 JSON、网络超时、权限不足 |
| 边界值 | 极端但合法的输入 | 空字符串、超长输入、零值、最大值 |
| 空值/nil | nil 指针、空 slice、空 map | `nil` 参数不 panic |
| 并发安全 | 多 goroutine 同时调用 | 共享状态无竞态（如适用） |
| 幂等性 | 重复调用结果一致 | 同一输入多次调用输出相同（如适用） |

*3.T3 mock 策略审查*

- mock 层级是否正确？应 mock 外部依赖（DB、HTTP、LLM），不应 mock 被测函数的内部实现细节
- mock 返回值是否覆盖了成功和失败两种情况？
- 是否存在"mock 了一切导致测试只验证了 mock 本身"的问题？
- mock 对象的行为是否与真实依赖一致？（如 mock DB 是否模拟了真实的错误码）

*3.T4 测试质量红线*

- 测试是否有有效断言？（禁止只调用不断言的"烟雾测试"）
- 断言是否验证了核心业务逻辑，而非仅检查"不报错"？
- 测试用例命名是否清晰表达了测试意图？（如 `TestParseLayout_EmptyInput_ReturnsError`）
- 是否使用了 table-driven tests 来覆盖多组输入？（Go 项目推荐）

**BVT（集成/端到端测试）方案**

*3.T5 端到端场景覆盖矩阵*

对本次变更涉及的业务流程，检查 BVT 是否覆盖了以下场景：

| 场景类型 | 检查项 | 说明 |
|----------|--------|------|
| 核心正向流程 | 主要业务路径的完整走通 | 如：上传文件 → 解析 → 提取 → 返回结果 |
| 关键分支流程 | 重要的条件分支 | 如：不同文件类型走不同处理管线 |
| 错误恢复流程 | 异常输入或依赖故障时的降级行为 | 如：模型服务超时 → 降级返回部分结果 |
| 回归场景 | 历史 bug 对应的复现路径 | 如果 PR 修复了 bug，必须有能 100% 复现该 bug 的 BVT |
| 并发场景 | 多请求并发时的正确性 | 如：并发提交同一文档不会数据错乱 |

*3.T6 断言质量审查*

- 断言是否精确验证了业务结果（字段值、数据条数、状态码 + 响应体）？
- 是否存在过度宽松的断言？常见问题：
  - `status_code == 200` 但不检查响应体内容
  - `len(result) >= 1` 代替 `len(result) == expectedCount`
  - 只检查 `error == nil` 不检查返回值
  - 使用 `Contains` 代替精确匹配（当精确匹配可行时）
- 断言失败时的错误信息是否足够定位问题？（是否包含 actual vs expected）

*3.T7 测试数据与环境管理*

- 测试数据是否自包含？是否依赖外部状态（特定数据库记录、特定文件、特定模型输出）？
- 测试间是否有数据隔离？一个测试的数据是否会污染另一个测试？
- 测试数据的 setup/teardown 是否完整？（创建的资源是否清理）
- 是否依赖特定的执行顺序？（测试应可独立运行、可乱序运行）
- 并发执行安全：多个 BVT 并行跑时是否会因共享资源（同一租户、同一文件名）而互相干扰？

*3.T8 回归验证（Bug Fix PR 专项）*

当 PR 修复了 bug 时，必须检查：
- 是否有专门的 BVT 用例能精确复现该 bug 的触发条件？
- 回归用例的断言是否直接验证了 bug 的修复效果（而非间接验证）？
- 回归用例是否在修复前会失败、修复后会通过？（即真正的回归测试）

**Eval（效果评估）方案** — 仅当 PR 涉及 LLM/VLM prompt、AI 调用链、模型切换、数据处理管线时审查
- 是否需要 eval 验证？变更是否影响 AI 输出质量（prompt 修改、模型替换、前/后处理逻辑变更）？
- eval 数据集是否充分？是否覆盖了多种输入类型（不同语言、不同格式、边界 case）？
- eval 指标是否明确？是否有量化的通过标准（准确率、召回率、格式合规率等），而非主观判断？
- 回归对比：是否与改动前的 baseline 做了对比？是否有 eval 结果数据支撑"不退化"的结论？
- 如果 PR 未包含 eval 但应该有：指出需要补充 eval 的场景和建议的评估方式

**测试补充建议输出格式**

当检测到测试遗漏时，必须按以下格式给出具体的补充建议，让开发者可直接 action：

```
🧪 测试补充建议：

1. [UT] `pkg/service.ParseLayout()` — 缺少单元测试
   - 建议补充场景：
     a. 正常 PDF 输入 → 返回正确 layout 结构
     b. 空输入 → 返回明确错误（非 panic）
     c. 超大页数输入 → 验证截断/限制逻辑
   - 建议 mock：mock `VLMClient` 接口，分别返回成功/失败/超时
   - 优先级：🔴 必须（核心业务函数无测试）

2. [BVT] 缺少文件类型 X 的端到端用例
   - 建议补充场景：上传 X 类型文件 → 完整处理 → 验证输出字段
   - 建议断言：检查 output.field_count == expected，而非仅 status 200
   - 优先级：🟡 建议（覆盖新增的处理分支）
```

输出格式：
- ✅ 方案合理的部分 + 理由
- 💡 可优化的建议 + 具体方案 + 预期收益
- 🧪 测试方案缺陷/缺失 + 建议补充的测试策略（使用上述补充建议格式）

#### 四、代码审查（逐文件 Review）

对每个变更文件进行审查，关注：
- 代码正确性：逻辑错误、边界条件、空指针、类型安全
- 代码风格：命名规范、函数长度、注释质量
- 错误处理：是否有遗漏的错误处理、错误信息是否清晰
- 可维护性：代码重复、魔法数字、硬编码
- 测试覆盖：关键路径是否有测试

输出格式（按严重程度分类）：
- 🔴 **必须修改**：会导致 bug 或严重问题
- 🟡 **建议修改**：代码质量或可维护性问题
- 🟢 **可选优化**：锦上添花的改进

每条 Review 包含：
```
文件：<file_path>
行号：<line_range>
问题：<description>
建议：<suggestion with code example if applicable>
```

**文档内交叉引用**

报告中经常出现跨章节引用（如"潜在风险"引用"代码审查"中的具体条目）。必须使用 Markdown 锚点链接，使引用可点击跳转：

1. 每条 Review 条目的标题前添加 HTML 锚点 `<a id="issue-N"></a>`（N 为全局编号）：
   ```markdown
   #### <a id="issue-1"></a>1. `pdf_v1.go` — `downloadImageBytes` 无响应体大小限制
   ```

2. 所有引用处使用 Markdown 链接格式 `[🔴 #N](#issue-N)`：
   ```markdown
   - ⚠️ HTTP 图片下载无响应体大小限制（见 [🔴 #1](#issue-1)）
   ```

这样读者在"潜在风险"章节看到引用时，可以直接点击跳转到"代码审查"中的详细描述。

#### 4.5、API / 配置变更清单

当 PR 涉及对外暴露的接口、配置项、数据结构变更时，必须单独列出，帮助调用方和运维快速评估影响。

**检查范围**（有则列，无则整章省略）：

**API 接口变更**
- 新增 / 删除 / 重命名的 HTTP / gRPC / WorkItem 接口
- 请求 / 响应字段的增删改（含类型变更、required 变更）
- 接口行为变更（如返回值语义变化、错误码变化）
- 向后兼容性判断：旧版调用方是否会 break？

**配置项变更**
- 新增 / 删除 / 重命名的配置项（YAML、环境变量、Feature Flag）
- 默认值变更
- 配置项的生效范围（全局 / 租户级 / 请求级）

**数据结构 / Schema 变更**
- 数据库 schema 变更（新增表、字段、索引）
- 消息队列 / 事件 schema 变更
- 文件格式变更（输入 / 输出）

**依赖变更**
- 新增外部依赖（系统命令、第三方服务、SDK）
- 版本升级的 breaking change

输出格式（表格）：

```markdown
| 类型 | 变更项 | 变更内容 | 向后兼容 | 备注 |
|------|--------|----------|----------|------|
| API | `moi:llm.extract.structured` | 合并原 `.advanced`，新增 `json_schema`/`files` 等字段 | ⚠️ 旧 `.advanced` 接口已删除 | 调用方需迁移 |
| 配置 | `extract_config.vl_model` | 新增，默认 `qwen3-vl-plus` | ✅ | 零值回退默认 |
| Schema | `StrikethroughVLMResponse` | 新增 Pydantic model | ✅ | 仅内部使用 |
```

每条变更附带：
- **向后兼容性**：✅ 兼容 / ⚠️ 需迁移 / 🔴 Breaking
- **迁移指引**：如果不兼容，说明调用方需要做什么

#### 五、潜在风险检查

逐项检查以下风险维度：

**并发安全**
- 共享资源是否有竞态条件
- 锁的粒度是否合理
- 是否有死锁风险
- channel/goroutine 是否有泄漏风险

**性能问题**
- 是否有 N+1 查询
- 是否有不必要的内存分配或拷贝
- 循环中是否有可提取的计算
- 数据库查询是否缺少索引
- 是否有可能的慢查询

**复杂度分析**
- 关键函数/算法的时间复杂度是否合理（标注 Big-O）
- 空间复杂度是否可优化（临时数据结构、缓冲区大小）
- 在大数据量场景下是否存在性能退化风险
- 是否有更优复杂度的替代算法

**成本问题**
- 是否有不必要的 API 调用（云服务、第三方）
- 资源是否及时释放（连接、文件句柄）
- 缓存策略是否合理
- 日志级别是否合适（避免生产环境过多 debug 日志）

**安全问题**
- 输入是否有校验
- 是否有 SQL 注入、XSS 等风险
- 敏感信息是否有泄漏风险
- 权限检查是否完整

**除幻（LLM/VLM Prompt 质量）** — 仅当 PR 涉及 LLM/VLM prompt 或 AI 调用链时审查

- **Prompt 覆盖率**：prompt 是否覆盖了所有关键场景和边界条件？
  - 输入变体：不同语言、不同格式、异常输入是否都有指令覆盖
  - 输出约束：是否明确约束了输出格式、禁止项、边界行为
  - 负面示例：是否提供了"不要做什么"的 few-shot negative examples
- **Prompt 稳定性**：对 LLM 输出随机性的容错能力
  - 输出解析是否有清洗/归一化逻辑（去代码块、JSON 包装等）
  - 是否有 fallback 策略应对 LLM 不遵守指令
  - 是否依赖精确格式（脆弱）vs 模糊匹配（健壮）
  - 多步调用中前一步幻觉是否会传播累积
- **可编程程度**：prompt 中有多少逻辑可以用确定性代码替代？
  - 能用正则/规则/代码做的事不要交给 LLM（格式校验、数值计算等）
  - LLM 输出后是否有程序化校验层（列数校验、schema 验证等）
  - prompt 中的硬编码值是否应该参数化
- **模型兼容性**：prompt 是否考虑了多模型适配？
  - 同一 prompt 是否需要在多个模型上运行？如果是，是否在不同模型上验证过效果？（不同模型对指令的遵循能力、输出格式偏好差异很大）
  - 是否有按模型分文件的模板机制？常见方式包括：按文件名区分（如 `layout_detection_qwen.jinja2`）或按目录区分（如 `./qwen3.5-27b/layout_detection.jinja2` 覆盖通用的 `./layout_detection.jinja2`）。如果当前只有一个模板但未来要支持多模型，是否预留了模板选择/路由逻辑？具体实现不限，关键是有没有这个分层意识。
  - prompt 中是否有模型特定的指令？（如 thinking mode、特定 system prompt 格式）这些指令换模型后是否会失效或产生副作用？

输出格式：
- 🧠 幻觉风险 + 具体 prompt 位置 + 改进建议
- ✅ 已检查无问题的维度

**省 Token（LLM/VLM 调用效率）** — 仅当 PR 涉及 LLM/VLM 调用时审查

- **执行性能**：
  - 串行 vs 并行？是否有可并行化的串行调用？
  - 总耗时与数据规模的关系（如 N 页 = O(N) 延迟）
- **Token 复杂度**（分析 token 消耗与数据规模 N 的关系）：
  - 输入 token：O(1) 固定 prompt vs O(N) 全量数据？
  - 输出 token：期望输出与数据规模的关系
  - 总 token = 调用次数 × 单次消耗（如 N 次 × O(N) = O(N²)）
  - 是否有截断/摘要策略将 O(N) 降为 O(1)
- **总体消耗**：
  - 与改动前相比总 token 增减？
  - 是否有冗余信息重复发送（如每次都发完整上下文）
  - prompt 模板是否可精简？是否可缓存/复用减少调用？

输出格式：
- 💰 Token 效率问题 + 具体位置 + 优化建议 + 预估节省
- ✅ 已检查无问题的维度

**快速迭代（可插拔性与演进能力）** — 评估变更是否有利于后续快速开发

- **可插拔**：核心策略/算法是否以可插拔方式实现？
  - 是否通过策略模式、接口抽象等方式解耦？能否不改主流程就替换实现？
  - LLM/VLM 调用是否有清晰的接口层？切换模型/provider 的成本如何？
  - 新增的处理步骤能否独立启用/禁用？
- **可开关**：关键行为是否支持配置化控制？
  - 新功能是否有 feature flag 或配置项控制启停？
  - 关键参数（截断行数、重试次数、阈值、模型选择等）是否可配置而非硬编码？
  - 是否支持灰度（按租户/文件类型等维度逐步开启）？
- **可替换**：当前实现能否低成本替换为更优方案？
  - 核心逻辑是否职责单一、边界清晰？能否独立演进不影响其他模块？
  - 数据结构和接口设计是否为未来扩展留有余地？
- **可测试**：新增逻辑是否易于单元测试？
  - 是否有难以 mock 的外部依赖耦合？
  - 纯函数 vs 有副作用的方法比例是否合理？
- **可调试**：出问题时能否快速定位？
  - 是否有足够的中间状态日志？prompt 和 LLM 响应是否可追溯？
  - 新旧方案能否方便地 A/B 对比？

输出格式：
- 🔧 迭代障碍 + 具体位置 + 改进建议
- ✅ 已检查无问题的维度

**违禁操作检查** — 检查代码中不应出现的硬编码或危险模式

- **超时硬编码**：超时时间是否直接写死在代码中（如 `time.Sleep(30 * time.Second)`、`timeout: 60000`）？
  - 超时值必须通过配置项、环境变量或常量定义，禁止在业务逻辑中硬编码
  - 检查 HTTP client timeout、gRPC deadline、数据库查询超时、重试间隔等所有超时场景
  - 如果是测试代码中的超时，可豁免但需标注

输出格式：
- 🚫 违禁操作 + 具体位置 + 应改为的方式（违禁操作一律为"必须修改"）
- ✅ 已检查无问题的维度

**测试实现质量** — 与"方案评审"中的测试方案梳理互补，此处聚焦代码级实现细节

*断言有效性*
- 已有测试是否因本次改动需要更新？更新后断言是否仍然有效？
- 断言是否验证了核心逻辑（而非仅检查副作用或日志输出）？
- 是否存在"永远通过"的断言？（如断言一个总为 true 的条件）

*测试代码 bug*
- expected 值是否正确？（常见：复制粘贴后忘改 expected）
- 是否有遗漏的 cleanup / teardown？（未关闭的连接、未删除的临时文件、未恢复的全局状态）
- 测试中的 goroutine 是否有泄漏风险？（如 `go func` 未等待完成）
- 错误路径测试是否真的触发了错误？（如 mock 配置错误导致走了 happy path）

*Flaky 风险*
- 是否有 timing 依赖？（`time.Sleep`、固定超时等待、依赖执行速度）
- 是否有 ordering 依赖？（依赖测试执行顺序、依赖 map 遍历顺序）
- 是否有环境依赖？（依赖特定端口、特定文件路径、特定环境变量）
- 并发测试是否有竞态？（多个测试并行时共享资源冲突）

*测试可维护性*
- 测试代码是否过度耦合实现细节？（实现重构后测试是否需要大量修改）
- 是否有大量重复的 setup 代码可提取为 helper？
- 测试失败时的错误信息是否足够定位问题？

输出格式：
- 🧪 测试实现问题 + 具体位置 + 修复建议
- ✅ 已检查无问题的维度

输出格式（通用风险）：
- ⚠️ 发现的风险 + 具体位置 + 修复建议
- ✅ 已检查无问题的维度

---

### 5. 输出最终报告

将所有维度的审查结果合并为一份完整报告，在终端直接输出。

### 6. 归档审查报告

审查报告输出后，同时将完整报告保存为 Markdown 文件到 `~/pr_review/` 目录备案：
- 文件名格式：`<repo>_PR<number>_<title>_<YYYYMMDD>.md`（例如 `matrixflow_PR8638_paddle-server-optimize_20260318.md`）
- repo 名从 PR URL 中提取（取最后一段，如 `matrixorigin/matrixflow` → `matrixflow`）
- title 从 PR 元数据中获取，去掉特殊字符，空格替换为 `-`，截断到 50 字符以内
- 如果目录不存在则自动创建
- 文件内容为完整的审查报告（与终端输出一致）

**重新 Review 时保留历史报告**：写入新报告前，检查 `~/pr_review/` 下是否已存在同名文件。如果存在，将旧文件重命名为 `_bakNNN` 格式保留（NNN 为三位数序号，从 001 递增），然后再写入新报告。重命名规则：去掉原文件的 `.md` 扩展名，追加 `_bakNNN.md`。如果已有 `_bakNNN.md`，则找到当前最大序号 +1。示例：
```bash
# 首次重新 review：
# matrixflow_PR8638_paddle-server-optimize_20260318.md
# → matrixflow_PR8638_paddle-server-optimize_20260318_bak001.md

# 再次重新 review（已有 _bak001.md）：
# matrixflow_PR8638_paddle-server-optimize_20260318.md
# → matrixflow_PR8638_paddle-server-optimize_20260318_bak002.md
```

### 7. 折叠历史审查评论（Hide Outdated）

发布新报告前，检查 PR 上是否已有当前用户发布的历史审查评论，如果有则通过 GraphQL `minimizeComment` 将其标记为 OUTDATED 折叠。

```bash
# 1. 获取当前用户名
CURRENT_USER=$(gh api user --jq '.login')

# 2. 获取 PR 所有评论的 node ID、作者、是否已折叠、body 前 200 字符
#    使用分页获取全部评论（每次 100 条）
gh api graphql -f query='
query($owner: String!, $repo: String!, $pr: Int!, $cursor: String) {
  repository(owner: $owner, name: $repo) {
    pullRequest(number: $pr) {
      comments(first: 100, after: $cursor) {
        pageInfo { hasNextPage endCursor }
        nodes {
          id
          author { login }
          isMinimized
          body
        }
      }
    }
  }
}' -f owner=<OWNER> -f repo=<REPO> -F pr=<PR_NUMBER>

# 3. 筛选：author.login == CURRENT_USER && !isMinimized && body 包含审查报告特征
#    审查报告特征：body 以 "# Code Review:" 开头，或包含 "## 〇、总结（TL;DR）"
#    对每条匹配的评论执行 minimize：
gh api graphql -f query='
mutation($id: ID!) {
  minimizeComment(input: {subjectId: $id, classifier: OUTDATED}) {
    minimizedComment { isMinimized }
  }
}' -f id=<COMMENT_NODE_ID>
```

注意：
- 只折叠当前用户自己的评论，不影响其他人的评论
- 只折叠未被折叠的评论（`isMinimized: false`）
- 通过 body 内容特征识别审查报告，避免误折叠普通讨论评论
- 如果有多条历史审查评论，全部折叠

### 8. 发布审查报告到 PR Comment

归档完成后，将完整审查报告作为 PR comment 发布：
```bash
gh pr comment <PR_NUMBER> --repo <OWNER/REPO> --body-file <归档的md文件路径>
```

## 审查原则

### 标准模式
1. **务实导向**：只提有价值的建议，不吹毛求疵
2. **给出方案**：每个问题都附带具体的修复建议或代码示例
3. **分清主次**：严重问题优先，风格问题次之
4. **理解意图**：结合 PR 目的和上下文来评审，避免脱离业务的纯技术挑剔
5. **中文输出**：报告使用中文

### 探索模式
1. **方案优先**：重点是"方案对不对"，而非"代码好不好"
2. **数据驱动**：关注方案是否能产出可评估的数据
3. **迭代友好**：关注方案是否为后续演进留好了路
4. **宽容细节**：不纠结代码风格、命名规范、微小性能问题
5. **标注 TODO**：对于"当前可以接受但生产化时必须解决"的问题，标注为 `📌 生产化 TODO`
6. **中文输出**：报告使用中文

## Gotchas

1. **大 PR diff 截断**：先用 `grep -n '^diff --git'` 定位文件边界，再按需 `sed -n` 读取，避免上下文溢出。
2. **自动生成文件**：swagger.json、docs.go 等跳过，仅确认存在。
3. **私有仓库**：`web_fetch` 无法访问私有仓库，必须使用 `gh` CLI。
4. **上下文不足时**：如果 diff 不足以判断问题，用 `gh api` 获取完整文件内容辅助判断。
